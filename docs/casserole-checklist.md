# CASSEROLE Checklist

Use this checklist before passing a request to anyone — a teammate, a contractor, a collaborator, or even yourself.

The goal is not to fill out a form. The goal is to make sure the request can actually travel.

---

## How to Use This

Work through each element. For each one, ask the question and answer it honestly. If you can't answer it, that's useful information — it means there's a gap to close before the request goes anywhere.

You don't need to write a paragraph for each item. A sentence or two is often enough. What matters is that the answer exists.

---

## The Checklist

---

### ✦ C — Clarity

> What exactly is being asked?

- [ ] The core ask is written out in plain language
- [ ] The expected output or deliverable is described
- [ ] There is no language that's vague by habit ("make it better," "just clean it up," "you know what I mean")
- [ ] Someone unfamiliar with the backstory could read this and understand what's needed

**Write it here:**
_What is being asked?_

---

### ✦ A — Alignment

> Does everyone agree on the goal?

- [ ] The goal has been stated, not assumed
- [ ] The person requesting and the person receiving understand what success looks like
- [ ] This request is consistent with broader priorities (or the deviation is intentional and noted)

**Write it here:**
_What is the goal?_

---

### ✦ S — Scope

> What's in, and what's out?

- [ ] What is included in this request is explicitly described
- [ ] What is excluded (or assumed to be excluded) is noted
- [ ] Any gray areas — things that might be assumed to be in scope but aren't — are addressed

**Write it here:**
_What is in scope? What is out of scope?_

---

### ✦ S — Safety

> What should not be touched, broken, or put at risk?

- [ ] Dependencies, integrations, or systems that could be disrupted are identified
- [ ] Relationships or agreements that constrain the approach are noted
- [ ] Anything that has caused problems before and should be treated carefully is flagged

**Write it here:**
_What must be protected?_

---

### ✦ E — Expectations

> What does "done" look like?

- [ ] The expected output format is described (document, working feature, verbal update, etc.)
- [ ] The quality bar is set — not just "good," but specifically what "good" means here
- [ ] Completion criteria are written out
- [ ] A deadline or timeframe is included, or its absence is intentional and noted

**Write it here:**
_What does done look like?_

---

### ✦ R — Reasonability

> Is the ask actually doable?

- [ ] The timeline is achievable given real constraints
- [ ] The required resources and access are available or accounted for
- [ ] The ask is within the realistic capability of the person receiving it
- [ ] Nothing has been overlooked that would make this impossible or unfair as written

**Write it here:**
_What makes this reasonable? What would make it unreasonable?_

---

### ✦ O — Ownership

> Who is responsible for what?

- [ ] The decision-maker is identified
- [ ] The person doing the work is identified
- [ ] Anyone who needs to review or approve is identified
- [ ] Anyone who should be kept in the loop (but not asked to act) is identified

**Write it here:**
_Who owns what?_

---

### ✦ L — Limits

> What are the hard constraints?

- [ ] Hard deadlines are noted
- [ ] Budget constraints (if any) are stated
- [ ] Tool, access, or platform constraints are noted
- [ ] Legal, compliance, or policy limits that apply are flagged

**Write it here:**
_What are the non-negotiables?_

---

### ✦ E — Explain Why

> What is the purpose behind this ask?

- [ ] The reason this request exists is stated
- [ ] The problem it solves (or the opportunity it serves) is described
- [ ] What happens if it isn't done — or isn't done well — is noted
- [ ] Any context that would change how someone approaches the work is included

**Write it here:**
_Why does this matter?_

---

## After the Checklist

Once you've worked through CASSEROLE, you should have enough to write a handoff that can actually be received.

If any element is incomplete, decide: is that a gap that needs to be closed before sending, or is it an acceptable unknown that should be flagged explicitly in the handoff itself?

A good handoff that acknowledges what it doesn't know is better than a confident handoff that hides it.
